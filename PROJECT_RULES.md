# PROJECT_RULES.md — BadBuddhas Bubbles

Telegram Mini App + веб для breathwork практик. Форк от BadBuddhas Breathwork с новым дизайном.
Стек: Next.js 14 (App Router) · TypeScript · Tailwind CSS · Supabase · Telegram Bot API · Vercel.

## Деплой и сервисы

- **Bot:** новый Telegram бот (токен в .env — TELEGRAM_BOT_TOKEN)
- **Vercel:** новый проект (отдельный от Breathwork)
- **Supabase:** ОБЩАЯ база с Breathwork (vnmjdzaszvatqeywbylw.supabase.co) — НЕ создавать новую!

---

## Архитектура

1. **UI-логика — только в `components/`.**
   Компоненты отвечают за рендер и пользовательские события. Никакой бизнес-логики.

2. **Бизнес-логика и работа с данными — только в `hooks/`.**
   Запросы к Supabase, трансформации данных, состояние — всё через хуки.

3. **Прямые вызовы Supabase/Telegram SDK — только в `lib/` и `hooks/`.**
   Импортировать `supabase`, `telegram` или SDK напрямую в `components/` запрещено.

4. **Новые TypeScript-типы — в `types/`.**
   Не определять типы inline внутри компонентов или хуков.

5. **API routes — в `app/api/`, один файл = один эндпоинт.**
   Каждый route-файл экспортирует ровно один HTTP-метод (или несколько для одного ресурса).

---

## Правила внесения изменений

6. **Перед любой правкой — сделать `git commit` текущего состояния.**
   Изменения начинаются с чистого или зафиксированного рабочего дерева.

7. **До и после правки запускать `npm run check`.**
   `npm run check` = `tsc --noEmit && next lint && next build`. Оба прогона должны быть зелёными.

8. **Рефакторинг — только по явному запросу.**
   Если задача не содержит слова «рефакторинг» — не переписывать код вокруг.

9. **Максимум 3 файла в одном изменении.**
   Если нужно трогать больше — разбить на последовательные шаги и согласовать.

10. **Не переименовывать файлы и функции без причины.**
    Переименование — отдельный шаг с явным обоснованием.

11. **Не менять форматирование за пределами затронутых строк.**
    Нет «заодно почищу отступы» — это засоряет diff.

---

## Как добавить новый экран

12. **Порядок действий:**
    1. Создать `app/<новый-экран>/page.tsx`
    2. Если нужны данные — создать хук `hooks/use<ИмяФичи>.ts`
    3. Если нужен переиспользуемый UI — создать `components/<ИмяКомпонента>.tsx`
    4. Если нужны новые типы — добавить в `types/`

---

## Как добавить новую фичу

13. **Сначала план, потом код.**
    План: какие файлы затрагиваются, что именно меняется, какие риски.
    Код пишется только после согласования плана.

14. **Минимальный diff.**
    Менять ровно столько, сколько нужно для задачи — не больше.

15. **После кода: `npm run check` + ручная проверка в браузере/Telegram.**

---

## Работа с Supabase

16. **Изменения схемы БД — только через файл миграции в `supabase/migrations/`.**
    Прямые изменения через Supabase Dashboard без миграции запрещены.

17. **Нумерация миграций: следующий номер `010_описание.sql`.**
    Формат: `NNN_короткое_описание_snake_case.sql`.

18. **После миграции — обновить `types/database.ts`.**
    Типы должны соответствовать актуальной схеме БД.

---

## Аутентификация (справка)

19. **Гибридная схема:**
    - В Telegram Mini App — авторизация через `telegram_id` (без пароля)
    - В браузере — авторизация через email/password
    - Один пользователь может иметь оба способа входа (linked-аккаунт)

20. **Провайдер аутентификации — `AuthProvider` (`components/AuthProvider.tsx`).**
    Текущий пользователь доступен через хук `useUser` (`hooks/useUser.ts`).
    Не обращаться к Supabase Auth напрямую из компонентов.
